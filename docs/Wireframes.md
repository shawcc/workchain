# MVP 低保真线框图草稿

> 说明：本文档是 `workgraph/wireframe_design.yaml` 的阅读视图，不是项目的主事实源。低保真原型的一等产物是线框图设计图谱。

## 1. 设计原则

MVP 原型不以文档编辑器作为默认主界面，而以“目标图谱工作台”作为默认形态。

核心原则：

- 用户不是写文档，而是在塑造工作图谱。
- Agent 不是聊天对象，而是在维护图谱、生成候选变更、执行任务。
- 关键推理都要记录 `f(x)`。
- 每个页面都必须明确创建或更新哪些图谱节点和关系。
- 任务完成不能只靠状态，必须有证据节点支撑。

## 2. 全局布局

建议 MVP 使用三栏工作台：

```text
┌─────────────────────────────────────────────────────────────┐
│ 顶部：确认门状态条 / 当前目标状态 / 下一步动作                │
├───────────────┬─────────────────────────────┬───────────────┤
│ 左侧目标导航   │ 中间主工作区                  │ 右侧 Agent 面板 │
│ 目标树         │ 目标、图谱、任务、证据          │ 候选变更        │
│ 状态筛选       │ 当前页面主体                    │ 推理记录        │
│ 待确认入口     │                               │ 待确认问题      │
└───────────────┴─────────────────────────────┴───────────────┘
```

常驻组件：

- 确认门状态条。
- 图谱面包屑。
- Agent 候选变更卡。
- 推理记录抽屉。

## 3. 页面一：目标入口页

目的：让 CEO 输入战略意图，并创建原始意图节点。

```text
┌─────────────────────────────────────────────────────────────┐
│ WorkGraph                                       新建目标      │
├───────────────┬─────────────────────────────┬───────────────┤
│ 最近目标       │ 你想让组织达成什么目标？        │ 推荐 Agent      │
│ - 东南亚增长   │ ┌─────────────────────────┐ │ 市场 Agent      │
│ - 降本增效     │ │ 我们要在东南亚市场拿下... │ │ 销售 Agent      │
│               │ └─────────────────────────┘ │ 财务 Agent      │
│ 目标模板       │ 上传背景材料                  │ 法务 Agent      │
│ - 营收目标     │ 目标类型：[营收增长 v]          │                 │
│ - 市场进入     │ [开始理解目标] [保存草稿]       │ 示例目标        │
└───────────────┴─────────────────────────────┴───────────────┘
```

关键字段：

- 原始意图。
- 背景材料。
- 目标类型。
- 参与 Agent。

关键按钮：

- 开始理解目标。
- 使用模板。
- 保存草稿。
- 从会议导入。

图谱变化：

- 创建 `raw_intent` 节点。
- 创建 `background_material` 节点。
- 创建 `derives_from` 关系。

异常态：

- 未输入目标。
- 材料上传失败。
- 无可用 Agent。

## 4. 页面二：目标理解工作台

目的：把原始意图展开为目标草案、假设、约束和待确认问题。

```text
┌─────────────────────────────────────────────────────────────┐
│ 状态：目标理解中 -> 目标待确认                              │
├───────────────┬─────────────────────────────┬───────────────┤
│ 原始意图       │ 目标草案                     │ Agent 分析      │
│ 东南亚 1 亿... │ 指标：新签美元收入             │ 市场 Agent      │
│               │ 目标值：100,000,000           │ 销售 Agent      │
│ 背景材料       │ 范围：印尼/泰国/越南           │ 财务 Agent      │
│ - 市场报告     │ 周期：未来 12 个月             │                 │
│ - 销售数据     │                               │ 候选变更        │
│               │ 假设 / 约束 / 成功标准          │ + 新增预算约束  │
│               │ 待确认问题                     │                 │
│               │ [提交目标确认] [重新分析]       │ 推理记录 f(x)   │
└───────────────┴─────────────────────────────┴───────────────┘
```

关键字段：

- 指标。
- 目标值。
- 市场范围。
- 时间周期。
- 业务范围。
- 负责人。
- 假设。
- 约束。
- 待确认问题。
- 成功标准。

关键按钮：

- 提交目标确认。
- 回答问题。
- 接受 Agent 候选变更。
- 要求 Agent 重新分析。
- 保存草案。

图谱变化：

- 创建 `goal_draft` 节点。
- 创建 `assumption` 节点。
- 创建 `constraint` 节点。
- 创建 `open_question` 节点。
- 创建 `reasoning_record` 节点。

异常态：

- 关键问题未回答。
- 约束冲突。
- 目标草案置信度低。

## 5. 页面三：小目标拆解页

目的：把确认后的目标拆成充分条件和阶段性小目标。

```text
┌─────────────────────────────────────────────────────────────┐
│ 状态：目标已确认 -> 小目标待确认                            │
├───────────────┬─────────────────────────────┬───────────────┤
│ 已确认目标     │ 小目标树                      │ Agent 解释      │
│ 1 亿美元收入   │ Goal                         │ 为什么这样拆？  │
│ 12 个月        │ ├─ 印尼市场收入主战场          │                 │
│               │ ├─ 泰国渠道试点                │ 风险列表        │
│ 拆解方式       │ ├─ 越南产品适配与合规           │ - 合规周期长    │
│ [市场] [漏斗]  │ ├─ 销售漏斗建设                │ - 渠道不确定    │
│ [职能] [时间]  │ └─ 交付与客户成功能力           │                 │
│               │ 支撑逻辑面板                   │ 推理记录 f(x)   │
│               │ [确认关键小目标] [重新拆解]     │                 │
└───────────────┴─────────────────────────────┴───────────────┘
```

关键字段：

- 小目标名称。
- 支撑逻辑。
- 指标。
- 目标值。
- 负责人。
- 依赖。
- 风险。
- 验收标准。

关键按钮：

- 确认关键小目标。
- 切换拆解方式。
- 重新生成方案。
- 编辑小目标。
- 标记关键小目标。

图谱变化：

- 创建 `sub_goal` 节点。
- 创建 `supports` 关系。
- 创建 `depends_on` 关系。
- 创建 `reasoning_record` 节点。

异常态：

- 小目标无法支撑总目标。
- 出现循环依赖。
- 高风险小目标没有负责人。

## 6. 页面四：工作图谱视图

目的：展示目标、假设、约束、小目标、任务、证据、异常和推理记录之间的关系。

```text
┌─────────────────────────────────────────────────────────────┐
│ 图谱视图                                                     │
├───────────────┬─────────────────────────────┬───────────────┤
│ 筛选           │         目标节点             │ 节点详情        │
│ [目标]         │            │ supports        │ 名称            │
│ [小目标]       │       小目标节点              │ 类型            │
│ [任务]         │            │ depends_on      │ 状态            │
│ [证据]         │         任务节点              │ 负责人          │
│ [异常]         │            │ proves          │ 关联证据        │
│               │         证据节点              │ 推理记录 f(x)   │
│ 保存视图       │                               │ [查看链路]      │
└───────────────┴─────────────────────────────┴───────────────┘
```

关键操作：

- 展开支撑链路。
- 查看推理记录。
- 按节点类型筛选。
- 生成阅读快照。
- 对比版本。

图谱变化：

- 更新节点。
- 更新关系。
- 创建快照。

异常态：

- 孤立任务。
- 缺少证据。
- 快照已过期。

## 7. 页面五：行动板

目的：承载任务生成、分配、执行、证据回填和阻塞处理。

```text
┌─────────────────────────────────────────────────────────────┐
│ 行动板                                                       │
├───────────────┬─────────────────────────────┬───────────────┤
│ 筛选           │ To Do / Doing / Blocked / Done │ 任务详情      │
│ 小目标         │ ┌────────┐ ┌────────┐       │ 支撑小目标      │
│ 负责人         │ │分析印尼│ │筛选渠道│       │ 依赖任务        │
│ 状态           │ │竞品定价│ │合作伙伴│       │ 验收标准        │
│               │ └────────┘ └────────┘       │ 证据要求        │
│ 阻塞任务       │ 依赖时间线                    │ Agent 执行      │
│               │ [从小目标生成任务]             │ [启动执行]      │
└───────────────┴─────────────────────────────┴───────────────┘
```

关键字段：

- 任务名称。
- 支撑的小目标。
- 负责人。
- 状态。
- 依赖。
- 优先级。
- 验收标准。
- 证据要求。

关键按钮：

- 从小目标生成任务。
- 启动 Agent 执行。
- 提交验收。
- 分配负责人。
- 上传证据。
- 标记阻塞。
- 升级为异常。

图谱变化：

- 创建 `task` 节点。
- 创建 `evidence` 节点。
- 创建 `proves` 关系。
- 更新任务状态。
- 创建 `exception` 节点。

异常态：

- 任务没有关联小目标。
- 任务完成但没有证据。
- 阻塞任务没有原因。
- 异常需要决策。

## 8. f(x) 在交互中的呈现

每个关键确认动作旁边都应有一个“推理记录”入口。

```text
推理记录：f(x) -> y

x 输入信息：
- 市场报告
- 销售漏斗数据
- 合规风险
- 预算上限

f(x) 推理过程：
- 识别目标要求：12 个月内形成可确认新签收入
- 比较三国市场信息：规模、渠道、合规周期、产品适配成本
- 判断越南合规周期对短期收入路径构成阻断
- 将确定性更高的印尼作为主路径，同时保留泰国低成本试点

y 阶段结论：
- 印尼提高权重
- 泰国走渠道试点
- 越南先做产品适配与合规准备
```

复盘时，系统可以判断：

- 是 `x` 不完整，导致判断错误；
- 是 `f(x)` 的某个推理步骤跳跃、遗漏或权重失真；
- 还是 `y` 执行失败，但推理过程本身合理。

这会让产品从“记录工作”进化为“校准组织推理过程”。

## 9. 人和 Agent 的协作模式

| 阶段 | 人类负责 | Agent 负责 | 图谱变化 |
| --- | --- | --- | --- |
| 目标入口 | 输入意图和背景 | 识别目标类型、建议 Agent | `raw_intent`、`background_material` |
| 目标理解 | 回答问题、确认约束 | 生成目标草案、假设和问题 | `goal_draft`、`assumption`、`constraint` |
| 小目标拆解 | 选择拆解路径 | 生成方案和风险解释 | `sub_goal`、`supports`、`depends_on` |
| 执行推进 | 授权和处理异常 | 生成任务、执行和回填证据 | `task`、`evidence`、`exception` |
| 复盘改进 | 判断推理过程哪里需要校准 | 对比推理链和后验结果 | `reasoning_record`、`retrospective_feedback` |

## 10. MVP 验收标准

- MVP 页面不能以文档编辑器作为默认主界面。
- 每个页面必须明确它会创建或更新哪些图谱节点和关系。
- 目标确认、小目标确认和任务验收必须展示推理记录入口。
- 行动板中的任务必须能追溯到小目标和总目标。
- 标记完成的任务必须有证据节点支撑。
