# 交互设计草稿：WorkGraph 人-Agent 协作办公系统

> 说明：本文档是 `workgraph/interaction_design.yaml` 的交互设计阅读视图，不是项目的主事实源。项目的一等交互产物是可推理交互图谱。

## 1. 交互设计核心判断

这个产品的交互不应该围绕“写文档、改文档、评审文档”展开，而应该围绕“塑造工作图谱”展开。

用户看到的不是一个空白文档，也不是一个聊天窗口，而是一个正在被人和 Agent 共同构建的目标图谱：

```text
原始意图 -> 目标 -> 假设/约束 -> 小目标 -> 任务 -> 证据 -> 验收 -> 反馈/修正
```

因此，交互设计的核心原则是：

- 用户不是在写文档，而是在塑造图谱。
- Agent 不是聊天对象，而是图谱维护者和执行者。
- 人类负责承诺，Agent 负责展开。
- 大段文字可以存在，但只能作为输入、解释或快照。
- 每次关键交互都应该落到图谱节点或关系变化上。
- 每次关键推理都应该记录为 `f(x)`，让输入信息、推理过程和阶段结论可复盘。

## 2. 产品整体壳层

### 2.1 左侧目标导航

用于按目标组织入口：

- 公司级目标。
- 事业部目标。
- 项目目标。
- 个人负责目标。
- 异常待处理目标。

它不是文档目录，而是目标图谱目录。

### 2.2 顶部模式切换

建议保留五个一级视图：

- 战略板：目标决策层主界面。
- 图谱视图：查看完整推理关系。
- 行动板：任务执行层主界面。
- 证据库：查看所有证据和来源。
- 复盘视图：从图谱生成阶段复盘。

### 2.3 Agent 协作侧栏

Agent 不应该以“聊天机器人”作为主形态，而应以“当前正在维护图谱的协作者”出现。

侧栏展示：

- Agent 正在分析什么。
- Agent 提出了哪些候选节点。
- Agent 提出了哪些候选关系。
- Agent 发现了哪些风险。
- Agent 需要人类确认什么。

### 2.4 确认门状态条

顶部或右侧展示当前目标状态：

```text
意图已输入 -> 目标理解中 -> 目标待确认 -> 小目标待确认 -> 执行中 -> 异常待决策 -> 已验收
```

确认门是产品信任机制的核心。

### 2.5 推理记录抽屉

在目标确认、小目标确认、任务生成、异常处理和验收判断旁边，展示本次推理的 `f(x)`：

- `x`：本次决策用到的信息、证据、假设、约束和上下文。
- `f(x)`：本次从输入信息到阶段结论的推理过程，包括前提选择、证据比较、约束处理和关键推理步骤。
- `y`：本次形成的目标版本、拆解方案、任务方案或偏差处理方案。

这个抽屉让复盘不再只看“结果好不好”，而是能看“当时的推理过程哪里做对了、哪里需要校准”。

## 3. 核心页面

### 3.1 目标入口页

目标：接收 CEO 或管理者的原始战略意图。

关键交互：

- 输入自然语言目标。
- 上传背景材料。
- 选择参与 Agent。
- 点击“开始理解目标”。

输入示例：

```text
我们要在东南亚市场拿下 1 亿营业额。
```

系统生成：

- 原始意图节点。
- 背景材料节点。
- 从原始意图到目标草案的推导关系。

### 3.2 目标理解工作台

目标：让 Agent 把原始意图展开成可确认目标。

页面分区：

- 左侧：原始意图和背景材料。
- 中间：结构化目标草案。
- 右侧：Agent 分析、待确认问题、假设和风险。

目标草案字段：

```yaml
metric: ""
target_value: ""
market_scope: ""
time_range: ""
business_scope: ""
owner: ""
assumptions: []
constraints: []
success_criteria: []
open_questions: []
```

用户操作：

- 回答澄清问题。
- 修改目标字段。
- 标记关键约束。
- 要求 Agent 重新分析。
- 提交目标确认。

### 3.3 目标确认页

目标：把目标草案升级为经确认的目标版本。

页面重点：

- 原始意图。
- 结构化目标。
- 与原始意图的差异。
- 关键假设。
- 关键约束。
- 高风险点。

操作：

- 确认目标。
- 驳回目标。
- 要求补充澄清。
- 生成目标版本。

确认后，系统创建：

- 目标节点。
- 目标版本。
- 确认记录节点。
- `approves` 关系。

### 3.4 小目标拆解页

目标：把总目标拆成充分条件和阶段性小目标。

页面重点：

- 总目标。
- 小目标树。
- 每个小目标的支撑逻辑。
- 小目标之间的依赖。
- 高风险假设。

拆解视角：

- 按市场拆解。
- 按漏斗拆解。
- 按职能拆解。
- 按时间阶段拆解。
- 按充分条件拆解。

操作：

- 查看 Agent 拆解方案。
- 对比多个拆解方案。
- 调整小目标权重。
- 删除或新增小目标。
- 确认关键小目标。

### 3.5 工作图谱视图

目标：展示目标到执行的完整推理结构。

图谱节点：

- 原始意图。
- 目标。
- 假设。
- 约束。
- 小目标。
- 任务。
- 证据。
- 验收。
- 异常。
- 决策记录。

图谱关系：

- 推导。
- 支撑。
- 依赖。
- 证明。
- 阻塞。
- 影响。
- 确认。
- 修订。

交互：

- 点击节点进入详情。
- 展开某个目标的支撑链路。
- 查看某个任务为什么存在。
- 查看某个证据证明了什么。
- 查看某个异常影响哪些目标。
- 生成阅读快照。

### 3.6 行动板

目标：承载任务生成、任务调度和执行推进。

视图：

- 看板视图。
- 任务依赖视图。
- Agent 负载视图。
- 阻塞列表。

任务卡片应展示：

- 任务名称。
- 支撑的小目标。
- 责任方。
- 状态。
- 依赖。
- 证据要求。
- 验收标准。

关键原则：

任务不是孤立事项。每个任务都必须能追溯到小目标和总目标。

### 3.7 任务执行页

目标：让 Agent 或人类执行任务，并留下可验收证据。

页面展示：

- 任务描述。
- 支撑的小目标。
- 输入要求。
- 执行日志。
- 输出结果。
- 证据。
- 验收标准。

操作：

- 启动 Agent 执行。
- 上传证据。
- 请求人工介入。
- 提交验收。
- 驳回并返工。

### 3.8 重大偏差处理页

目标：当执行推翻假设或影响目标达成时，回流决策层。

页面展示：

- 偏差描述。
- 影响范围。
- 受影响目标和小目标。
- 相关证据。
- Agent 给出的处理选项。
- 决策记录。

操作：

- 调整目标。
- 调整小目标。
- 调整资源。
- 暂停任务。
- 继续执行。

### 3.9 复盘视图

目标：从图谱生成复盘，而不是人工写复盘文档。

复盘结构：

- 原目标是什么。
- 关键假设是什么。
- 哪些假设成立。
- 哪些假设被推翻。
- 哪些任务完成。
- 哪些证据支持完成。
- 哪些决策发生过修订。
- 下一轮目标建议。

## 4. 状态机

### 4.1 目标状态

```text
raw_intent_created
-> understanding_in_progress
-> goal_pending_approval
-> goal_approved
-> sub_goals_pending_approval
-> execution_ready
-> executing
-> exception_pending_decision
-> achieved
-> archived
```

### 4.2 任务状态

```text
pending
-> assigned
-> running
-> blocked
-> evidence_pending
-> acceptance_pending
-> done
```

也可能进入：

```text
rejected -> running
```

## 5. Agent 行为规则

### 5.1 候选修改而非直接覆盖

Agent 对目标、小目标、任务和验收的修改，默认生成候选变更，等待有权限的人确认。

### 5.2 输出必须落图谱

Agent 不能只输出长文本建议。它的输出必须转化为：

- 节点。
- 关系。
- 证据。
- 风险。
- 决策选项。
- 待确认问题。

### 5.3 不确定性显式化

Agent 对低置信度判断必须创建：

- 假设节点。
- 风险节点。
- 待确认问题节点。

### 5.4 执行必须有证据

Agent 标记任务完成前必须回填证据，并说明证据如何证明任务完成。

### 5.5 关键推理必须记录 f(x)

Agent 在生成目标确认稿、小目标拆解、任务方案、偏差处理建议和验收判断时，必须记录：

- 输入信息 `x`。
- 推理过程 `f(x)`。
- 阶段结论 `y`。
- 置信度。
- 后续复盘反馈。

## 6. 协作模拟

### 6.1 第一步：CEO 输入目标

CEO 输入：

```text
我们要在东南亚市场拿下 1 亿营业额。
```

系统不是生成一篇战略文档，而是创建：

- `raw_intent` 节点。
- 原始表达记录。
- 背景材料入口。

Agent 开始识别缺失字段。

### 6.2 第二步：Agent 提出澄清问题

目标理解 Agent 提出：

- 1 亿营业额的币种是什么？
- 时间周期是本财年、未来 12 个月还是某个季度？
- 东南亚市场包含哪些国家？
- 营业额口径是 GMV、收入还是回款？
- 是否包含既有客户续费？

这些问题不是聊天记录，而是 `open_question` 节点。

### 6.3 第三步：CEO 回答问题

CEO 回复：

```text
目标是未来 12 个月，美元收入，覆盖印尼、泰国、越南，只计算新签收入。
```

系统更新：

- `goal_draft.metric = 新签美元收入`
- `goal_draft.time_range = 未来 12 个月`
- `goal_draft.market_scope = 印尼、泰国、越南`
- 相关 `open_question` 标记为已解决。

### 6.4 第四步：多 Agent 讨论

市场 Agent：

```text
印尼可能是主战场，泰国适合渠道合作，越南需要产品本地化。
```

销售 Agent：

```text
需要拆成线索、商机、转化率、客单价和回款周期。
```

财务 Agent：

```text
需要明确收入确认和预算约束。
```

法务 Agent：

```text
数据合规和本地合同模板是前置风险。
```

系统落图谱：

- 市场假设节点。
- 销售漏斗假设节点。
- 财务约束节点。
- 合规风险节点。
- 风险对目标的 `impacts` 关系。

### 6.5 第五步：生成目标确认稿

系统生成结构化目标：

```yaml
metric: 新签美元收入
target_value: 100000000
time_range: 未来 12 个月
market_scope:
  - 印尼
  - 泰国
  - 越南
constraints:
  - 合规通过后才能规模化获客
  - 预算需在后续确认
success_criteria:
  - 12 个月累计新签收入达到 1 亿美元
  - 每个国家有可追踪销售漏斗
```

CEO 看到的是一个可确认目标，而不是一份方案文档。

### 6.6 第六步：CEO 确认目标

CEO：

```text
确认这个目标版本。预算上限先按 1500 万美元建模。
```

系统创建：

- `goal` 节点。
- `approval_record` 节点。
- `approves` 关系。
- 新增预算约束节点。

目标进入 `goal_approved` 状态。

### 6.7 第七步：Agent 拆解小目标

Agent 生成小目标：

- 建立三国市场进入策略。
- 形成可规模化获客渠道。
- 完成本地化产品与合规准备。
- 建立销售漏斗并达成阶段收入。
- 建立交付和客户成功能力。

每个小目标必须说明如何支撑 1 亿美元收入目标。

### 6.8 第八步：CEO 确认小目标

CEO：

```text
先确认，但印尼权重提高，泰国以渠道试点，越南先做产品适配。
```

系统更新：

- 印尼相关小目标权重上调。
- 泰国目标标记为渠道试点。
- 越南目标从收入主目标调整为产品适配和合规准备。
- 生成小目标确认记录。

### 6.9 第九步：Agent 生成任务图

系统生成任务：

- 分析印尼目标客户和竞品定价。
- 筛选泰国渠道合作伙伴。
- 评估越南产品本地化缺口。
- 设计销售漏斗指标和看板。
- 制定 1500 万美元预算分配方案。

每个任务都有：

- 支撑的小目标。
- 依赖关系。
- 责任 Agent 或人。
- 验收标准。
- 证据要求。

### 6.10 第十步：执行与证据回填

Agent 或人类执行任务后，回填证据：

- 市场分析报告链接。
- 渠道伙伴名单。
- 本地化需求清单。
- 销售漏斗 BI 看板。
- 预算模型表。

任务不是因为“状态改成完成”而完成，而是因为证据证明其满足验收标准。

### 6.11 第十一步：重大偏差回流

系统发现：

```text
越南合规周期预计超过 9 个月，影响 12 个月收入目标。
```

系统创建：

- `exception` 节点。
- 对越南小目标的 `impacts` 关系。
- 偏差报告。
- 决策选项。

进入重大偏差确认门。

### 6.12 第十二步：CEO 决策

CEO：

```text
越南收入目标下调，资源转向印尼；越南保留产品适配和合规准备。
```

系统更新：

- 越南小目标。
- 印尼资源配置。
- 相关任务优先级。
- 决策记录。
- 修订关系。

这不是一次聊天结束，而是图谱发生了可追溯变化。

## 7. 第一版原型建议

MVP 原型建议优先做 5 个页面：

- 目标入口页。
- 目标理解工作台。
- 小目标拆解页。
- 工作图谱视图。
- 行动板。

暂缓：

- 复盘中心完整能力。
- 高级图谱编辑器。
- 多行业模板。
- 复杂企业权限。

## 8. 仍需确认的问题

- CEO 首次使用时应该看到空白输入框，还是行业目标模板？
- Agent 协作过程默认展示多少细节？
- 图谱视图是主界面，还是战略板/行动板的底层可展开视图？
- 哪些任务允许 Agent 自动执行，哪些必须人工确认？
