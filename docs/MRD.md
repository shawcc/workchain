# MRD：目标驱动的人-Agent 协作办公系统

> 说明：本文档是 `workgraph/product_workgraph.yaml` 的市场需求阅读视图，不是项目的主事实源。项目的一等产物是可推理工作图谱。

## 1. 文档信息

## 2. 一句话定义

WorkGraph 是一套面向企业管理与复杂项目推进的人-Agent 协作办公系统，将 CEO 或管理者的战略意图转化为可确认的目标、可推理的小目标、可执行的任务图，并通过 Agent 协作完成执行、证据回填和目标验收。它不以文档作为核心产物，而以可推理工作图谱作为组织工作的第一性载体。

## 3. 市场背景

### 3.1 办公模式正在从“人读文档”转向“人-Agent 共推理”

传统办公软件以文档、聊天、会议和表格为核心载体，主要服务于人类之间的信息传递与共识形成。随着 Agent 具备理解、规划、执行和工具调用能力，办公系统不再只是记录信息，而要能够驱动工作推进。

新的办公系统需要回答：

- 管理者的模糊战略意图如何变成明确目标？
- 目标如何被拆解为充分条件和阶段性小目标？
- 小目标如何继续拆解为可执行任务？
- Agent 如何在执行过程中记录证据、反馈异常、推动验收？
- 人类管理者在哪些关键节点介入确认，而不是被任务细节淹没？

### 3.2 现有工具的核心矛盾

现有办公工具通常分散在多个系统中：

- 聊天工具承载讨论，但状态不可追踪。
- 文档工具承载说明，但难以执行和验收。
- 项目管理工具承载任务，但缺少目标与决策上下文。
- BI/数据工具承载指标，但不能自动连接战略、任务和行动。
- AI 聊天工具能生成内容，但容易停留在对话层，缺少组织级工作闭环。

这导致企业在战略执行中存在明显断层：

```text
战略意图 -> 会议讨论 -> 文档沉淀 -> 任务拆解 -> 人工推进 -> 结果复盘
```

每一环都依赖人工理解、人工转译和人工追踪，信息损耗大，决策与执行脱节。

## 4. 市场机会

### 4.1 从“AI 助手”到“AI 工作系统”

当前大量 AI 产品仍以助手形态存在，主要解决写作、总结、问答和局部自动化问题。企业真正需要的是可承载组织协作的工作系统：

- 不只是回答问题，而是推动目标达成。
- 不只是生成文本，而是生成可执行工作图。
- 不只是单个 Agent 工作，而是多个 Agent 围绕目标协作。
- 不只是任务管理，而是连接目标、假设、任务、证据和验收。

### 4.2 从“自然语言办公”到“可推理办公”

人类自然语言适合表达意图、价值判断和模糊方向，但不适合作为复杂工作的最终载体。Agent 时代的办公表达需要从大段文本迁移到结构化推理对象：

- 目标可确认。
- 条件可拆解。
- 任务可调度。
- 证据可追踪。
- 结果可验收。
- 异常可回流决策。

### 4.3 从“文档产物”到“工作图谱产物”

在传统办公中，文档是主要产物：战略文档、方案文档、会议纪要、项目计划、复盘报告。文档适合人阅读，但不适合作为 Agent 执行、推理、调度和验收的底层对象。

WorkGraph 的核心产物应是可推理工作图谱：

- 原始意图被保存为图谱的来源节点，而不是最终产物。
- 目标、假设、约束、小目标、任务、证据、验收和异常都成为图谱节点。
- 支撑关系、依赖关系、因果关系、版本关系和确认关系成为图谱边。
- 文档只能是图谱的阅读视图、导出格式或汇报快照，而不是系统中的一等工作对象。

### 4.4 从“拍脑袋工作”到“推理驱动工作”

WorkGraph 的基本假设是：不管是人还是 Agent，任何工作都不应凭直觉孤立发生，而应来自一段可解释、可审计、可复盘的推理过程。

每个工作单元都应遵循同一套原模型：

```text
理解目标 -> 确认目标 -> 识别假设和约束 -> 拆解目标 -> 形成行动 -> 执行与回填 -> Review 与反馈
```

不同岗位、层级和 Agent 的差异不在于工作逻辑不同，而在于它们负责的图谱层级不同：

- 高层负责人处理更抽象、更上游、更高影响面的目标、约束和决策。
- 中层负责人处理跨团队目标拆解、资源配置、依赖协调和异常回流。
- 执行者和专项 Agent 处理更细粒度的任务、证据、执行动作和局部 review。
- 所有人和 Agent 的工作都必须能向上找到来源，并向下解释其服务的更大目标。

因此，WorkGraph 不是把公司工作拆成一堆独立任务，而是维护一条不能断裂的推理链路：任何行动都有出处，任何结论都有推理，任何失败都能反馈回上游重新安排。

### 4.5 与 OKR 的关系

WorkGraph 与 OKR 有相似性，但产品定位不是 OKR 工具。OKR 主要表达目标和关键结果，WorkGraph 表达的是目标从自然语言被理解、被确认、被拆解、被执行、被证明和被 review 的完整推理链路。

两者可以这样对应：

- Objective 对应 WorkGraph 中的 Goal，即被理解和确认后的目标对象。
- Key Results 对应 WorkGraph 中的 Target、Success Criteria 和 Evidence Requirement，即目标值、成功标准和证据要求。
- Initiative 或 Task 对应 WorkGraph 中的 Work Unit 或 Task，但上级视角里的 Task 到下一级负责人那里会成为新的 Goal。
- OKR Check-in 对应 WorkGraph 中的 Review，但 Review 不只是更新进度，而是可能触发重新讨论、重新确认或重新拆解。

因此，WorkGraph 可以承载 OKR，但它比 OKR 多了目标解释、假设、约束、推理记录、来源追溯、证据验收和异常回流。它关注的不是“把 OKR 写下来”，而是“让任何工作都能说明为什么做、怎么拆、如何证明、失败后如何反馈”。

## 5. 目标用户

### 5.1 核心用户

- CEO / 创始人：提出战略目标，确认目标理解和关键拆解路径。
- 事业部负责人：承接公司级目标，拆解业务目标，监控执行。
- 项目负责人 / PMO：维护目标树、任务图、依赖和风险。
- 职能负责人：围绕市场、销售、产品、运营、财务、人力等子目标组织 Agent 和人协作。

### 5.2 协作参与者

- 专项 Agent：市场分析 Agent、销售策略 Agent、产品 Agent、运营 Agent、财务 Agent、法务 Agent、数据分析 Agent 等。
- 人类专家：在关键判断、资源协调、价值取舍和审批节点介入。
- 执行系统：CRM、ERP、BI、项目管理、代码仓库、文档库、工单系统、投放平台等。

## 6. 典型场景

### 6.1 战略目标落地

CEO 表达：“我们要在东南亚市场拿下 1 亿营业额。”

系统需要完成：

- 理解目标口径：时间周期、市场范围、营业额定义、币种、业务线、资源边界。
- 组织 Agent 讨论：市场机会、竞品格局、渠道策略、产品适配、预算和风险。
- 生成目标确认稿：提交给 CEO 确认。
- 拆解小目标：例如市场进入、渠道建设、销售转化、产品本地化、品牌获客、合规准备。
- 再次确认拆解路径：CEO 或高管确认。
- 生成任务图：分配给 Agent 或人类团队执行。
- 持续跟踪：用证据和指标判断目标是否达成。

### 6.2 跨部门项目推进

当一个目标涉及多个部门时，系统将目标拆为子目标和任务依赖，自动识别阻塞项、责任主体和决策缺口。

### 6.3 经营复盘

系统基于目标树、任务执行记录、证据和指标变化，生成复盘结构：

- 原目标是什么？
- 当时的关键假设是什么？
- 哪些任务完成了？
- 哪些证据支持目标达成？
- 哪些假设被推翻？
- 哪些决策需要修正？

### 6.4 目标链路持续 Review

当上层目标被拆解到下层行动后，系统需要持续检查依赖项和执行结果。如果某个下游任务未完成、证据不足、假设被推翻或约束发生变化，它不应只停留在任务层失败，而应沿着图谱关系向上反馈，触发目标、拆解路径、资源配置或行动计划的重新 review。

典型链路：

```text
下游任务失败 / 证据不足 / 依赖阻塞
-> 影响的小目标被标记
-> 上游负责人收到决策待办
-> 重新讨论目标、假设和约束
-> 形成新的行动计划或重新拆解
```

这让企业从“结果不好再开会复盘”转变为“执行信号实时回流到推理链路中”。

## 7. 用户痛点

### 7.1 决策侧痛点

- 战略目标经常是自然语言表达，缺少结构化澄清。
- 管理者确认的是文档而不是可执行目标模型。
- 目标拆解依赖少数管理者经验，难以复用和审计。
- 决策依据、假设和约束容易散落在会议、聊天和文档中。

### 7.2 执行侧痛点

- 任务与原始目标脱节，团队只知道做什么，不知道为什么。
- 任务完成缺少证据标准，容易出现“状态完成但目标未达成”。
- 跨团队依赖不透明，阻塞发现滞后。
- 执行中发现目标不合理时，缺少回流决策机制。

### 7.3 AI 使用痛点

- AI 聊天容易生成大段建议，但无法自然进入执行闭环。
- 多 Agent 协作缺少共同目标模型，容易各自生成局部答案。
- Agent 输出缺少人类确认门，组织无法放心授权。
- Agent 执行缺少证据链，结果可信度不足。

## 8. 产品机会点

### 8.1 以“目标决策层 + 任务执行层”重构办公

产品底层工作模型采用两层：

- 目标决策层：负责意图理解、目标澄清、假设约束、小目标拆解和人类确认。
- 任务执行层：负责任务生成、Agent 调度、执行推进、证据回填、验收判断和异常回流。

但用户进入产品时不应首先面对流程步骤，而应进入一个以“我的可见工作空间”为中心的信息结构：

- 工作图谱：默认入口，展示用户有权限看到的整个工作图谱，并支持从任意节点逐层向下展开。
- 关注的工作：聚合用户收藏、关注或系统判断为重要的目标、任务、节点和变更。
- 决策待办：只承载真正需要用户确认、取舍、授权或处理的事项。
- 知识库：沉淀与工作相关的材料、证据、背景和复盘记录，并受权限控制。

这种结构让产品心智从“按流程操作系统”转向“围绕我可见图谱工作的空间”，降低高频用户在复杂流程中反复跳转的成本。

### 8.2 以“确认门”建立人机协作信任

关键节点必须由人确认：

- 目标确认门：确认系统对战略意图的理解。
- 小目标确认门：确认拆解路径是否符合战略意图。
- 重大偏差确认门：当执行偏离假设、资源或目标时，回到决策层。

### 8.3 以“工作图谱”替代文档产物

核心工作载体从自然语言文档变成目标-条件-任务-证据图谱：

```text
目标 -> 小目标/充分条件 -> 任务 -> 执行状态 -> 证据 -> 验收 -> 反馈
```

这意味着产品不应鼓励用户“写一份方案文档”作为终点，而应引导用户持续维护一张可推理、可执行、可验收的工作图谱。方案、纪要、复盘可以由图谱自动生成，但不反过来成为工作系统的主对象。

### 8.4 以“推理过程”改进组织判断力

WorkGraph 的另一个关键机会是把组织推理表达为可复盘、可校准的函数过程：

```text
f(x) -> y
```

- `x` 是当时获得的信息，包括数据、证据、假设、约束和上下文。
- `f(x)` 是完整推理过程，包括前提选择、证据比较、约束处理、推理步骤和结论形成。
- `y` 是阶段性推理产物，包括目标草案、小目标拆解、任务选择、资源配置或偏差处理方案。

传统复盘通常只看结果好坏，难以判断是信息不足、前提失真、推理跳跃还是执行失败。WorkGraph 记录每次关键推理的输入、过程和阶段结论，使企业能够长期校准自己的推理过程。

推理过程不只存在于高层战略讨论，也存在于每个下游工作单元。一个任务为什么存在、为什么由某个 Agent 执行、为什么需要某类证据、为什么失败后要回流，都应该能在图谱中找到上游推理来源。

### 8.5 以“权限内可见图谱”降低组织协作风险

企业工作图谱天然跨层级、跨部门、跨系统。用户需要向下展开自己负责或参与的工作，也可能需要向上追溯目标来源、决策依据和上游约束，但并不是所有上游信息都应对所有人可见。

WorkGraph 的机会在于把权限控制内化到图谱浏览和知识访问中：

- 用户看到的是“我有权限看到的整个工作图谱”，而不是全公司无边界总图。
- 当用户向上追溯且无权限时，系统应隐藏具体内容，只提示存在权限边界。
- 收藏、关注、决策待办和知识库都必须继承图谱节点、证据和材料的权限。
- Agent 在执行和总结时也只能使用当前用户或任务授权范围内的信息。

这能让企业在保留图谱化协作优势的同时，避免敏感战略、组织关系和经营数据被过度暴露。

## 9. 竞品与替代方案

### 9.1 项目管理工具

代表：Jira、Asana、Monday、飞书项目、Notion 项目管理。

优势：

- 任务、状态、责任人、看板成熟。

不足：

- 通常从任务出发，不从战略目标和充分条件出发。
- 缺少 Agent 自动理解、拆解、执行和证据回填。
- 目标、假设、决策和任务之间关联弱。

### 9.2 文档与知识库工具

代表：Notion、飞书文档、Confluence。

优势：

- 适合沉淀信息、说明背景、达成共识。

不足：

- 文档主要供人阅读，不天然可执行。
- 逻辑结构、任务依赖和验收标准需要人工维护。

### 9.3 AI 助手 / AI Agent 平台

代表：ChatGPT、Claude、Coze、Dify、各类企业 Agent 平台。

优势：

- 理解自然语言能力强，能生成方案和执行部分工具调用。

不足：

- 多停留在对话或工作流层。
- 缺少组织级目标确认、任务图、证据链和治理机制。

## 10. 差异化定位

WorkGraph 的核心差异不是“更会聊天的 AI”，而是“以目标为根的可推理工作系统”。

差异点：

- 从 CEO 意图开始，而不是从任务列表开始。
- 将所有人和 Agent 的工作统一为同一个推理驱动工作模型。
- 保证目标、拆解、任务、证据、review 和反馈形成不可断裂的链路。
- 默认呈现用户权限内的完整工作图谱，而不是碎片化页面或单一项目列表。
- 将关注工作、决策待办和知识库作为图谱派生视图，而不是独立割裂模块。
- 用目标确认门和小目标确认门控制授权边界。
- 用充分条件拆解目标，而不是简单头脑风暴任务。
- 用证据链验收任务，而不是只看状态。
- 用异常回流连接执行与决策，而不是执行失败后人工复盘。
- 支持多 Agent 围绕同一个目标模型协作。
- 以可推理工作图谱作为第一性产物，而不是以文档作为第一性产物。
- 记录关键推理的 `f(x)`，让组织能够复盘并校准推理过程。

## 11. 商业目标

### 11.1 短期目标

- 验证高管目标到执行任务的闭环是否成立。
- 在 3-5 个典型经营场景中形成可复用模板。
- 证明 Agent 能减少目标澄清、任务拆解和进度追踪的人力成本。

### 11.2 中期目标

- 成为企业战略执行和跨部门项目推进的 AI 工作台。
- 与企业现有系统集成，形成目标到业务数据的闭环。
- 沉淀行业目标拆解模板和 Agent 协作范式。

### 11.3 长期目标

- 成为企业级“可推理办公操作系统”。
- 从工具层进入企业经营决策与执行基础设施。

## 12. 成功指标

### 12.1 产品使用指标

- 目标创建数。
- 工作图谱展开深度和节点访问频次。
- 工作节点来源可追溯率。
- 下游异常向上回流处理率。
- 关注节点 / 收藏工作数量。
- 决策待办处理时长。
- 知识库权限内检索成功率。
- 目标确认率。
- 小目标确认率。
- 自动生成任务采纳率。
- Agent 执行任务完成率。
- 任务证据回填率。
- 异常回流处理率。

### 12.2 业务价值指标

- 目标澄清时间降低比例。
- 战略到任务拆解时间降低比例。
- 跨部门阻塞发现提前量。
- 管理者会议和人工追踪时间降低比例。
- 目标达成率提升。

### 12.3 信任指标

- CEO / 负责人对目标理解稿的首次通过率。
- 小目标拆解的人工修改率。
- Agent 输出被采纳率。
- 执行证据被验收通过率。

## 13. 市场风险

- 企业管理者可能习惯文档和会议，不愿切换到结构化目标模型。
- Agent 自动拆解质量不足会影响信任。
- 不同行业的目标拆解逻辑差异较大，通用模板难度高。
- 企业数据和系统权限复杂，影响执行闭环。
- 如果权限边界表达不清，用户会误以为图谱信息缺失或系统不完整。
- 如果产品过度复杂，用户会认为是建模工具而非办公工具。

## 14. 产品原则

- 人类负责关键承诺，Agent 负责结构化和执行推进。
- 产品底层心智保持两层：目标决策层和任务执行层。
- 所有人和 Agent 的工作都必须由推理产生，而不是孤立拍脑袋产生。
- 每个工作节点必须能向上追溯来源目标、假设、约束或决策。
- 每个下游失败、阻塞或证据不足都必须能向上触发 review。
- 产品入口心智保持四类：工作图谱、关注的工作、决策待办、知识库。
- 工作图谱默认只展示用户有权限看到的内容，向上追溯遇到无权限内容时必须明确提示边界。
- 关注、待办和知识库都是工作图谱的派生视图，不应形成新的信息孤岛。
- 系统内部保留三阶段：意图理解、显性推理、可执行推理。
- 所有任务必须能追溯到目标或小目标。
- 所有完成必须有证据。
- 所有重大偏差必须回流决策。
- 所有 review 必须回写图谱，并可能触发重新理解目标、重新确认目标或重新拆解。
- 不用大段自然语言或文档作为最终工作载体，但保留原始表达作为意图来源。
- 文档不是产品内的一等产物，只能作为可推理工作图谱的导出视图、汇报快照或阅读层。
