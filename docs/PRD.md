## 2. 产品概述

WorkGraph 是一套以目标为根的人-Agent 协作办公系统。它将管理者的自然语言战略意图转化为可确认的目标对象，再拆解为小目标、任务、执行状态、证据和验收结果，最终形成从决策到执行的闭环。产品不以文档作为核心产物，而以可推理工作图谱作为组织工作的第一性载体。

WorkGraph 的底层假设是：任何人和任何 Agent 的工作都应该根据推理产生，而不是孤立拍脑袋决定。每个工作单元都必须能说明它理解了什么目标、基于哪些假设和约束、如何拆解出行动、执行结果如何证明、失败时如何反馈回上游。

因此，系统不是简单管理任务，而是维护一条完整不断裂的目标推理链路：

```text
理解目标 -> 确认目标 -> 识别假设和约束 -> 目标拆解 -> 行动计划 -> 执行证据 -> Review 反馈
```

企业中不同角色的差异不在于工作模型不同，而在于负责的层级不同。高层处理更抽象、更上游的目标和决策；中层处理跨团队拆解、依赖和资源；执行者和 Agent 处理更细粒度的任务、证据和局部反馈。所有层级必须共享同一套推理链路。

产品一级结构采用两层：

- 目标决策层：负责意图理解、目标澄清、小目标拆解和人类确认。
- 任务执行层：负责任务生成、Agent 调度、执行推进、证据回填、验收判断和异常回流。

产品对用户暴露的一级信息结构采用四个主入口：

- 工作图谱：默认入口，展示用户有权限看到的整个工作图谱，支持从任意节点一层一层向下展开。
- 关注的工作：展示用户收藏、关注或系统认为重要的目标、节点、任务和变更。
- 决策待办：展示真正需要用户确认、取舍、授权或处理的事项。
- 知识库：展示权限范围内的材料、证据、背景、复盘和图谱派生知识。

四个入口都不是独立信息孤岛，而是同一张权限化工作图谱的不同视图。

系统内部保留三种推理状态：

- 隐性推理：承载自然语言、直觉、类比、偏好和模糊判断。
- 显性推理：沉淀目标、假设、约束、充分条件和小目标。
- 可执行推理：生成任务、依赖、状态、工具调用、证据和验收。

## 3. 产品目标

### 3.1 用户目标

- CEO 能用自然语言提出战略目标，并快速获得结构化目标理解稿。
- 管理者能确认目标和拆解路径，而不是亲自维护所有任务细节。
- Agent 能围绕同一个目标模型协作，而不是各自生成孤立建议。
- 执行团队能看到每个任务为什么存在，以及如何证明完成。
- 组织能追踪目标从提出、确认、拆解、执行到验收的全过程。
- 用户能从自己的权限范围内看到完整工作图谱，并理解哪些上游信息因权限不可见。
- 用户能快速进入自己关注的工作、需要处理的决策和可访问的知识。

### 3.2 产品目标

- 将战略目标转化为结构化、可确认、可执行的工作图。
- 通过确认门建立人类授权边界。
- 通过证据链提高 Agent 执行可信度。
- 通过异常回流让执行结果反向修正目标和假设。
- 让每个工作单元都能向上追溯来源目标、推理依据、假设和约束。
- 让下游执行失败、依赖阻塞和证据不足能触发上游 review 和重新拆解。
- 通过四个主入口降低复杂图谱的使用门槛。
- 通过权限化图谱浏览保护上游目标、敏感知识和跨部门信息。

### 3.3 非目标

- v0.1 不做完整企业 IM 替代。
- v0.1 不做通用项目管理工具的全部能力。
- v0.1 不承诺自动替代高管决策。
- v0.1 不追求全行业通用模板，优先验证战略目标落地场景。
- v0.1 不把方案文档、会议纪要或复盘文档作为核心工作产物，文档仅作为图谱导出和阅读视图。

## 4. 核心概念

### 4.1 目标节点 Goal Node

目标节点是产品里唯一的核心工作对象。`Goal`、`SubGoal`、`Work Unit` 本质上都是同一种对象，只是处在不同层级、不同视角、不同责任人手里。

```text
上级视角：这是一个 SubGoal / 下游工作
当前负责人视角：这是我的 Goal
系统抽象：它始终是一个 Goal Node
```

因此，系统不应把 Goal、SubGoal、WorkUnit 设计成三套对象，而应统一为 `Goal Node`。每个 Goal Node 都包含目标解释、目标值、假设、约束、成功标准、负责人、来源链路、推理记录、执行动作、证据和 review。

Goal Node 可以递归拆解：

```text
Goal Node
-> Child Goal Node
-> Child Goal Node
-> ...
```

它的关键含义是：任何一个工作节点都是目标化的，而不是孤立任务。

### 4.2 目标值 Target

Goal Node 中的量化靶点或结果要求。例如“明年营收 1 亿”里的 `营收 1 亿` 是 Target，而“实现明年经营增长”才是 Goal Node。

Target 通常包含：

- 指标名称：如营收、留存率、交付周期、成本。
- 目标值：如 1 亿、提升 20%、降低 30%。
- 口径说明：如是否含税、是否包含存量收入、统计范围是什么。
- 时间范围：如明年、Q1、未来 6 个月。

### 4.3 目标解释 Goal Interpretation

对原始自然语言目标的结构化理解，是从 Raw Intent 到 Goal Node 的推理产物。

例如用户说“明年营收一个亿”，系统不能直接把它当成可执行目标，而要解释：

- 这里的营收口径是什么？
- 覆盖哪个市场、产品线或业务线？
- 时间范围是否是自然年、财年还是未来 12 个月？
- 是否包含存量收入、渠道收入或一次性收入？
- 当前资源、预算和组织能力是否支撑该目标？

Goal Interpretation 不是最终目标，而是提交给负责人确认的目标理解稿。确认后，系统才生成正式 Goal Node 版本。

### 4.4 子目标 Child Goal Node

子目标不是一种新实体，而是某个 Goal Node 的下级 Goal Node。它必须能解释自己如何支撑上级 Goal Node。

当一个上级 Goal Node 被拆给下一级负责人时，它在上级视角里叫“子目标”或“下游工作”，但在下一级负责人视角里就是自己的 Goal Node。因此系统应允许任意 Child Goal Node 继续进入同一套流程：理解目标、确认目标、识别假设和约束、继续拆解、执行与 review。

### 4.5 假设 Assumption

目标拆解所依赖但尚未完全验证的前提。

Assumption 说明“为什么我们认为这个拆解路径可能成立”。例如东南亚市场仍有增长空间、现有产品能适配当地客户、渠道伙伴能在 6 个月内建立有效转化。

### 4.6 约束 Constraint

执行过程中不可违反的条件，如预算、时间、合规、品牌、安全、资源上限等。

Constraint 说明“行动不能突破什么边界”。例如预算不超过 300 万、不能新增 20 人团队、必须 6 个月内验证、必须符合当地合规要求。

### 4.7 成功标准 Success Criteria

用于判断 Goal Node 是否成立和完成的标准。Success Criteria 可以包含量化指标、证据要求、质量标准和验收条件。

在 OKR 语境中，Success Criteria 与 Key Results 接近，但 WorkGraph 中的 Success Criteria 必须连接证据和验收，而不只是写一个结果指标。

### 4.8 执行动作 Action / Task

为达成某个 Goal Node 而生成的可执行动作，包含责任方、依赖、状态、验收标准和证据要求。

Task 不应被理解为产品的核心对象。很多时候，上级视角里的 Task，到了下一个 owner 那里就是他的 Goal Node。比如 CEO 的 Goal Node 是“明年营收 1 亿”，拆给销售负责人的是“建立东南亚渠道体系”；对 CEO 来说这是一个下游动作或子目标，对销售负责人来说它就是自己的 Goal Node。

因此 Task 是 Goal Node 的执行态或行动清单，而不是另一个与 Goal Node 平级的核心抽象。

### 4.9 证据 Evidence

证明任务完成或目标推进的材料，可以是数据、文件、系统日志、链接、用户确认、实验结果等。

### 4.10 确认门 Approval Gate

需要人类负责人明确确认的关键节点，包括目标确认、小目标确认和重大偏差确认。

### 4.11 可推理工作图谱 Reasonable Work Graph

系统中的第一性工作产物，由节点和关系构成：

- 节点：原始意图、目标、假设、约束、小目标、任务、证据、验收、异常、决策记录、review 记录。
- 关系：支撑、依赖、因果、版本、确认、阻塞、证据证明、异常影响、拆解来源、反馈回流。

文档不是一等对象。方案、纪要、周报、复盘报告只能从图谱中生成，作为人类阅读视图、对外汇报快照或归档格式。

### 4.11.1 工作单元 Work Unit

工作单元是系统里最小的可推理工作对象，可以是目标、小目标、任务、研究、决策、证据收集、异常处理或 review。每个工作单元都必须包含：

- 来源：它来自哪个上级目标、假设、约束、决策或异常。
- 目标理解：当前工作要解决什么问题，服务哪个更大目标。
- 假设与约束：当前行动依赖哪些前提，受到哪些边界限制。
- 拆解结果：它向下拆成哪些子目标、任务或行动计划。
- 执行证据：完成或失败的依据是什么。
- Review 结果：是否需要回流上游、重新拆解或调整目标。

### 4.11.2 完整推理链路 Reasoning Chain

完整推理链路是从上游目标到下游行动，再从执行反馈回到上游 review 的连续关系。系统必须保证：

- 任意一个工作单元都能向上追溯到来源目标。
- 任意一个目标都能向下查看它如何被拆解为行动。
- 任意一个执行失败、依赖阻塞或证据不足都能向上反馈到受影响的目标或决策。
- 链路中断时，系统必须提示缺少来源、缺少支撑逻辑或缺少证据。

### 4.11.3 层级责任 Work Level

企业中不同人的差异主要是负责的图谱层级不同，而不是工作逻辑不同：

- 高层负责人负责公司级目标、关键假设、资源约束和重大方向确认。
- 中层负责人负责业务目标拆解、跨团队依赖、资源协调和异常回流。
- 执行者负责具体任务、局部判断、证据回填和执行反馈。
- Agent 可以参与任何层级，但必须遵守该层级的权限、目标和确认门。

### 4.12 推理过程 Reasoning Process

推理办公中的关键推理可以表达为：

```text
f(x) -> y
```

- `x`：当时获得的信息，包括数据、证据、背景、约束、假设和上下文。
- `f(x)`：从输入信息到阶段结论的完整推理过程，包括前提选择、证据比较、约束处理、推理步骤和结论形成。
- `y`：阶段性推理产物，包括目标草案、拆解方案、任务选择、资源配置或偏差处理方案。

系统需要记录每次关键推理的 `f(x)` 过程。复盘时，组织不只判断结果好坏，还要判断输入信息是否充分、前提是否成立、推理步骤是否跳跃、结论是否被证据验证。长期积累后，企业的推理过程可以持续校准。

### 4.13 与 OKR 的关系

WorkGraph 与 OKR 相似，但不是 OKR 工具。两者关系可以这样理解：

| OKR 概念 | WorkGraph 对应概念 | 差异 |
|----------|-------------------|------|
| Objective | Goal | Goal 不只是目标标题，还包含目标解释、来源链路、假设、约束和推理记录。 |
| Key Results | Target / Success Criteria | WorkGraph 要求 KR 连接证据、验收标准和执行反馈。 |
| Initiative / Task | Work Unit / Task | 上级的 Task 可以成为下级的 Goal，并继续拆解。 |
| Check-in | Review | Review 会触发上游重新讨论、重新确认或重新拆解，而不只是更新进度。 |

OKR 主要表达“目标是什么、结果是什么”。WorkGraph 进一步表达“目标为什么成立、怎么被理解、基于什么假设和约束、如何拆解、由谁继续推理、如何证明完成、失败后如何回流”。

### 4.14 关注工作 Focused Work

用户主动收藏、关注，或系统根据职责、风险、更新频率和决策影响自动推荐的重要工作集合。关注工作可以是目标、小目标、任务、证据、异常、推理记录或知识条目。

### 4.15 决策待办 Decision Inbox

需要用户明确做出承诺性动作的事项集合，包括目标确认、小目标确认、权限授权、重大偏差处理、验收判断和关键资源取舍。普通状态更新、信息通知和执行日志不应进入决策待办。

### 4.16 权限化知识 Permissioned Knowledge

与工作图谱节点关联的材料、证据、背景、复盘和外部系统信息。知识库必须继承图谱节点、证据来源和企业系统的权限规则，用户只能检索和查看自己有权限访问的内容。

### 4.17 Review 机制 Review Loop

Review 是执行信号回到推理链路的机制。当下游任务未完成、证据不足、依赖阻塞、假设被推翻或约束变化时，系统必须识别受影响的上游目标和决策，并触发重新讨论、重新确认或重新拆解。

Review 不是事后写复盘文档，而是图谱中的正式工作单元，会改变目标状态、任务计划、假设可信度或资源安排。

## 5. 用户角色

### 5.1 CEO / 最终决策者

权限：

- 创建公司级目标。
- 审阅目标理解稿。
- 确认或驳回目标版本。
- 确认关键小目标拆解。
- 处理重大偏差。

关注点：

- 系统是否正确理解战略意图。
- 拆解路径是否符合经营判断。
- 执行是否真实推进目标。

### 5.2 目标负责人

权限：

- 承接目标。
- 补充约束、资源和背景。
- 确认非公司级小目标。
- 调整任务优先级。
- 处理执行阻塞。

### 5.3 Agent 编排者

可以是系统角色或高级用户。

权限：

- 配置参与目标讨论的 Agent。
- 指定 Agent 能访问的工具和数据。
- 调整 Agent 分工和任务边界。

### 5.4 专项 Agent

职责：

- 参与目标理解和讨论。
- 提出假设、风险、拆解建议。
- 执行分配任务。
- 回填证据和异常。

### 5.5 普通协作者

职责：

- 查看与自己相关的任务。
- 提供人工输入和确认。
- 上传证据。
- 处理 Agent 无法完成的任务。

## 6. 核心流程

### 6.1 主流程

```text
CEO 输入战略意图
-> 人和 Agent 共同理解目标
-> 充分讨论假设、约束、不清楚的地方和成功标准
-> 形成目标理解稿和阶段结论
-> 负责人确认目标
-> 拆解为下一层目标或行动计划
-> 下一层继续按同一套逻辑理解、确认、拆解和执行
-> 执行并回填证据
-> Review 执行结果、依赖和假设
-> 成功则继续向下推进，失败则向上回流重新讨论和安排
```

### 6.2 决策层流程

```text
意图输入
-> 目标澄清
-> 假设识别
-> 约束识别
-> 多方推理讨论
-> 成功标准定义
-> 目标理解稿生成
-> 人类确认
-> 小目标拆解
-> 小目标确认
```

### 6.3 执行层流程

```text
小目标输入
-> 任务生成
-> 依赖分析
-> 责任分配
-> 执行调度
-> 状态更新
-> 证据回填
-> 验收判断
-> 异常回流
```

### 6.4 单个工作单元流程

任何层级的工作单元都必须复用同一套流程：

```text
接收上游来源
-> 理解当前目标
-> 补齐问题、假设和约束
-> 形成阶段结论
-> 拆解为下游工作
-> 执行或交给下游执行
-> 收集证据与状态
-> Review 是否达成目标
-> 必要时反馈上游重新推理
```

### 6.5 Review 回流流程

```text
下游任务未完成 / 依赖阻塞 / 证据不足 / 假设失效
-> 系统识别受影响的上游目标和行动计划
-> 生成 review 记录和决策待办
-> 相关负责人、人和 Agent 重新讨论目标、假设和约束
-> 维持原计划、调整资源、重排任务或重新拆解
-> 更新图谱版本并继续执行
```

Review 回流必须保留原始链路和新链路，避免执行问题被局部消化而无法影响上游决策。

## 7. 信息架构

### 7.1 顶层导航

- 工作图谱：默认 Tab，展示用户有权限看到的整个工作图谱，支持从任意节点向下逐层展开，也支持向上追溯可见的目标、决策和上下文。
- 关注的工作：第二个 Tab，展示用户收藏、关注或系统判断为重要的工作节点，帮助用户快速回到高价值工作。
- 决策待办：第三个 Tab，展示真正需要用户确认、取舍、授权或处理的事项，不混入普通通知。
- 知识库：第四个 Tab，展示用户权限范围内的材料、证据、背景、复盘和知识条目。

顶层导航原则：

- 四个 Tab 都从同一张可推理工作图谱派生，不创建彼此割裂的数据源。
- 工作图谱是默认入口，其他 Tab 是图谱的个人化筛选视图。
- 用户无权限查看的上游节点、知识或证据不展示内容，只展示必要的权限边界提示。
- Agent 生成摘要、推荐和待办时必须遵守当前用户权限。

### 7.2 工作图谱页

模块：

- 可见根节点和当前节点。
- 向下展开的子目标、任务、证据、异常和推理记录。
- 向上追溯的可见目标、约束和决策记录。
- 工作单元的来源、支撑逻辑、依赖和下游影响范围。
- 无权限上游信息的边界提示。
- 节点详情、关系说明和最近变化。
- 从节点进入关注、决策、知识或执行动作的入口。

### 7.3 关注的工作页

模块：

- 我收藏的工作。
- 我关注的工作。
- 系统推荐我关注的重要变化。
- 高风险或高频更新节点。
- 按目标、任务、证据、异常、知识分类筛选。
- 取消关注、置顶、提醒频率和进入图谱上下文。

### 7.4 决策待办页

模块：

- 待我确认的目标、小目标和任务验收。
- 待我处理的重大偏差。
- 待我授权的权限、工具调用和 Agent 行动。
- 每个待办的来源节点、影响范围、证据、Agent 建议和截止时间。
- 已处理决策记录和后续影响。

### 7.5 知识库页

模块：

- 权限内材料、证据、背景、复盘和外部系统链接。
- 与当前工作图谱节点的关联关系。
- 搜索、筛选、标签和来源追溯。
- 无权限知识的边界提示。
- 从知识条目跳转回相关图谱节点。

## 8. 功能需求

### 8.1 目标创建

用户故事：

作为 CEO，我希望用自然语言输入战略目标，使系统自动理解并生成结构化目标草案。

功能要求：

- 支持自然语言输入目标。
- 支持上传背景材料，如市场报告、经营数据、竞品分析。
- 支持选择目标类型，如营收目标、增长目标、市场进入、产品发布、降本增效。
- 系统自动抽取关键字段：市场、指标、目标值、时间周期、业务范围、约束、负责人。
- 系统保留原始表达，不覆盖人类意图来源。

验收标准：

- 输入一句自然语言战略目标后，系统能生成结构化目标草案。
- 草案必须展示缺失字段和待澄清问题。
- 原始输入必须可追溯。

### 8.1.1 图谱化原则

功能要求：

- 所有目标创建结果必须进入可推理工作图谱。
- 系统不得只生成一份静态目标说明文档作为产物。
- 目标理解稿应表现为目标节点、假设节点、约束节点、成功标准节点和待确认问题节点。
- 所有自然语言说明必须能映射到图谱节点或作为节点备注存在。

验收标准：

- 目标创建完成后，系统必须生成图谱节点和关系。
- 用户可以从图谱生成目标说明文档，但不能只保存文档而不保存图谱。
- 图谱节点更新后，相关文档视图必须能重新生成或提示过期。

### 8.2 目标理解与澄清

用户故事：

作为 CEO，我希望 Agent 在提交目标前先充分讨论并列出问题，使我确认的是清晰目标而不是模糊口号。

功能要求：

- 多 Agent 根据各自角色对目标进行分析。
- 支持人和 Agent 围绕同一目标进行结构化推理讨论，而不是沉淀为会议纪要或大段文档。
- 系统生成澄清问题，如时间范围、口径、资源边界、风险偏好。
- 系统识别隐含假设。
- 系统识别可能冲突的约束。
- 系统输出目标理解稿。
- 系统把讨论结论转化为图谱节点、关系、假设、约束和待确认问题。

目标理解稿字段：

```yaml
goal:
  raw_expression: ""
  normalized_description: ""
  metric: ""
  target_value: ""
  market_scope: ""
  time_range: ""
  owner: ""
  assumptions: []
  constraints: []
  success_criteria: []
  open_questions: []
```

验收标准：

- 目标理解稿必须包含目标字段、假设、约束、成功标准和待确认问题。
- Agent 讨论结论必须能追溯到参与 Agent 的观点。
- 讨论过程必须形成推理记录，而不是只保存聊天流水或会议纪要。
- CEO 可以对字段逐项修改或确认。

### 8.3 目标确认门

用户故事：

作为 CEO，我希望在系统推进执行前确认目标版本，避免 Agent 误解战略意图。

功能要求：

- 支持确认、驳回、要求补充澄清。
- 目标确认后生成不可变版本号。
- 所有后续小目标和任务必须关联目标版本。
- 目标变更时系统提示影响范围。

验收标准：

- 未确认目标不能进入任务执行层。
- 目标确认记录包含确认人、时间、版本和备注。
- 目标变更后必须生成新版本。

### 8.4 小目标拆解

用户故事：

作为目标负责人，我希望系统把总目标拆解为若干小目标，并说明每个小目标如何支撑总目标。

功能要求：

- Agent 基于目标、假设和约束生成小目标树。
- 每个小目标必须包含支撑逻辑。
- 每个小目标必须说明它来自哪一个上游目标、假设、约束或 review 结论。
- 支持按时间、业务模块、漏斗环节、组织职能、充分条件等方式拆解。
- 支持展示小目标之间的依赖关系。
- 支持标记关键小目标。
- 支持将行动计划继续拆解为下一层工作单元，并复用同一套理解、确认、拆解、执行和 review 流程。

小目标字段：

```yaml
sub_goal:
  id: ""
  description: ""
  parent_goal_id: ""
  support_logic: ""
  metric: ""
  target_value: ""
  owner: ""
  dependencies: []
  risks: []
  acceptance_criteria: []
```

验收标准：

- 每个小目标必须关联上级目标。
- 每个小目标必须有支撑逻辑。
- 系统必须标出拆解中的高风险假设。
- 系统必须能从任意小目标向上追溯到来源，并向下查看其拆解结果。

### 8.5 小目标确认门

用户故事：

作为 CEO 或目标负责人，我希望确认关键小目标，确保执行路径没有偏离战略。

功能要求：

- 支持逐项确认小目标。
- 支持批量确认低风险小目标。
- 支持要求 Agent 重新拆解。
- 支持对小目标设置确认责任人。

验收标准：

- 未确认的关键小目标不能生成正式任务。
- 确认记录可追溯。
- 重新拆解必须保留历史版本。

### 8.6 任务生成

用户故事：

作为执行负责人，我希望系统基于确认后的小目标生成任务图，减少人工拆解成本。

功能要求：

- 基于小目标生成任务列表。
- 自动识别任务依赖。
- 自动建议责任 Agent 或人类角色。
- 自动生成验收标准和证据要求。
- 支持人工编辑任务。
- 每个任务必须带有来源链路，说明它服务哪个上游目标和哪段推理结论。

任务字段：

```yaml
task:
  id: ""
  description: ""
  supports_sub_goal_id: ""
  owner_type: "agent | human | team"
  owner_id: ""
  dependencies: []
  status: "pending | running | blocked | done | rejected"
  priority: "low | medium | high | critical"
  input_requirements: []
  expected_output: ""
  acceptance_criteria: []
  required_evidence: []
```

验收标准：

- 每个任务必须关联至少一个小目标。
- 每个任务必须有验收标准。
- 每个任务必须有责任方。
- 每个任务必须能向上追溯到目标、假设、约束或 review 结论。

### 8.7 Agent 调度与执行

用户故事：

作为目标负责人，我希望 Agent 能根据任务图自动执行任务，并在需要人工判断时发起请求。

功能要求：

- 支持将任务分配给不同 Agent。
- 支持 Agent 调用外部工具和企业系统。
- 支持任务状态自动更新。
- 支持 Agent 在遇到权限、数据缺失、判断不确定时请求人工介入。
- 支持任务执行日志。
- Agent 执行前必须读取任务来源、目标、约束和证据要求。
- Agent 输出必须回写图谱，不能只输出自然语言建议。

验收标准：

- Agent 执行必须记录输入、操作、输出和时间。
- Agent 无法继续执行时必须进入 blocked 状态并说明原因。
- 关键工具调用必须可审计。
- Agent 执行结果必须关联来源任务和证据节点。

### 8.8 证据回填

用户故事：

作为管理者，我希望看到任务完成的证据，而不是只看到状态变成完成。

功能要求：

- 支持 Agent 自动上传证据。
- 支持人工上传证据。
- 支持证据类型标记：数据、文档、链接、截图、日志、审批、实验结果。
- 支持证据与任务、小目标和目标关联。
- 支持证据可信度标记。

验收标准：

- 没有满足证据要求的任务不能自动验收通过。
- 证据必须可追溯到来源。
- 证据删除或变更必须记录。

### 8.9 验收判断

用户故事：

作为目标负责人，我希望系统判断任务是否真的支撑小目标，而不是只完成了表面动作。

功能要求：

- 系统根据验收标准和证据判断任务完成情况。
- 支持自动验收、人工验收和混合验收。
- 支持验收不通过并要求返工。
- 支持小目标级验收。
- 支持总目标进度判断。
- 验收不通过时必须判断是否影响上游目标、假设或拆解路径。

验收标准：

- 验收结果必须包含依据。
- 验收不通过必须说明缺口。
- 小目标验收必须基于关联任务和指标。
- 验收不通过且影响上游目标时必须生成 review 记录或决策待办。

### 8.10 异常回流

用户故事：

作为 CEO，我希望执行中发现重大偏差时，系统能自动回到决策层，而不是继续错误执行。

功能要求：

- 系统识别重大偏差：目标不可达、资源不足、假设被推翻、合规风险、关键任务阻塞。
- 自动生成偏差报告。
- 发起重大偏差确认门。
- 支持调整目标、小目标、任务、资源或约束。
- 记录决策变更原因。
- 支持将下游任务失败、依赖阻塞或证据不足沿图谱关系回流到受影响上游目标。
- 支持基于回流结果重新理解目标、重新确认目标或重新拆解行动计划。

验收标准：

- 重大偏差不能只停留在任务层，必须回流目标决策层。
- 偏差报告必须包含影响范围和建议动作。
- 决策变更后必须更新关联任务。
- 回流后必须保留原链路、新链路和变更原因。

### 8.10.1 推理记录与推理过程校准

用户故事：

作为 CEO 或目标负责人，我希望系统记录关键推理当时基于什么信息、经历了什么推理过程、形成了什么阶段结论，并在复盘时指出哪些推理步骤有效、哪些前提失真、哪些结论跳跃，从而持续提升组织推理质量。

功能要求：

- 关键推理必须生成推理记录节点。
- 推理记录必须包含输入信息 `x`、推理过程 `f(x)`、阶段结论 `y`。
- 输入信息应关联证据、数据、背景材料、假设、约束和上下文。
- 推理过程应记录前提选择、证据比较、约束处理、关键步骤、模型判断和人工偏好。
- 输出结果应关联目标、小目标、任务、资源配置或偏差处理方案。
- 复盘时系统应比较推理过程、阶段结论和实际结果，标记信息缺失、前提失真、推理跳跃和过程校准建议。

验收标准：

- 目标确认、小目标确认、重大偏差处理必须生成推理记录。
- 每条推理记录必须能追溯输入信息和输出结果。
- 复盘报告必须能按推理链路展示哪些 `f(x)` 过程被验证，哪些需要校准。

### 8.11 Agent 协作室

用户故事：

作为管理者，我希望看到 Agent 之间如何讨论目标和拆解方案，以便判断输出是否可信。

功能要求：

- 展示参与 Agent 列表和角色。
- 展示关键观点、分歧和结论。
- 支持将讨论内容折叠为决策摘要。
- 支持查看 Agent 输出依据。
- 支持人工追问。

验收标准：

- Agent 讨论不能只保存聊天流水，必须生成结构化摘要。
- 分歧必须被标记。
- 进入目标理解稿的结论必须可追溯。

### 8.12 复盘中心

用户故事：

作为管理者，我希望系统基于目标和执行证据自动生成复盘，帮助组织学习。

功能要求：

- 自动生成目标复盘报告。
- 对比目标值、阶段值和实际结果。
- 总结关键假设是否成立。
- 总结任务完成情况和证据。
- 总结重大偏差和决策调整。
- 输出下一轮建议。

验收标准：

- 复盘必须引用目标、任务、证据和决策记录。
- 复盘结论必须区分事实、判断和建议。

### 8.13 工作图谱浏览

用户故事：

作为用户，我希望第一个 Tab 就能看到我有权限访问的整个工作图谱，并能从当前节点一层一层向下展开工作。

功能要求：

- 默认展示用户权限范围内的工作图谱。
- 支持节点逐层展开和收起。
- 支持从节点向上追溯可见的目标、决策、约束和上下文。
- 向上追溯遇到无权限节点时，不展示节点内容，只展示权限边界提示。
- 支持从任意节点进入详情、关注、决策、知识和执行入口。

验收标准：

- 用户进入产品默认看到工作图谱 Tab。
- 点击有子节点的图谱节点后可以展开下一层。
- 无权限上游内容不能泄露标题、正文、证据或敏感元信息。

### 8.14 关注的工作

用户故事：

作为用户，我希望快速看到我收藏或关注的重要工作，避免每次都从整张图谱重新定位。

功能要求：

- 支持用户关注或收藏目标、小目标、任务、证据、异常、推理记录和知识条目。
- 支持系统根据职责、风险、更新频率和决策影响推荐关注项。
- 支持置顶、取消关注、提醒频率设置和分类筛选。
- 关注列表中的每个条目必须能跳回图谱上下文。

验收标准：

- 用户可以从图谱节点添加或移除关注。
- 关注页只展示用户有权限查看的条目。
- 关注条目更新时展示最新变化和来源节点。

### 8.15 决策待办

用户故事：

作为用户，我希望只在第三个 Tab 看到真正需要我决策的事项，而不是被普通通知和执行状态淹没。

功能要求：

- 聚合目标确认、小目标确认、重大偏差处理、验收判断、权限授权和关键资源取舍。
- 每个待办展示来源节点、影响范围、证据依据、Agent 建议、截止时间和决策选项。
- 支持同意、驳回、要求补充信息、转交和延后处理。
- 普通状态更新、Agent 日志和仅供阅读的信息不得进入决策待办。

验收标准：

- 决策待办必须关联至少一个图谱节点。
- 决策完成后生成决策记录并更新相关图谱节点。
- 无权限查看依据的用户不能看到该待办的敏感上下文。

### 8.16 知识库

用户故事：

作为用户，我希望在第四个 Tab 搜索和浏览与工作相关的知识，但只能看到我有权限访问的内容。

功能要求：

- 支持检索材料、证据、背景、复盘、外部系统链接和图谱派生知识。
- 知识条目必须关联一个或多个图谱节点。
- 支持按来源、类型、目标、时间、权限范围和可信度筛选。
- 对无权限知识只展示权限边界，不展示内容摘要。
- 支持从知识条目跳转回相关工作图谱节点。

验收标准：

- 知识库搜索结果必须经过权限过滤。
- 每个可见知识条目必须展示来源和关联图谱节点。
- Agent 生成知识摘要时不能使用当前用户无权限查看的内容。

### 8.17 Review 与重新拆解

用户故事：

作为负责人，我希望下游任务失败、依赖阻塞或假设失效时，系统能把问题反馈到上游目标链路中，而不是让问题停留在局部任务里。

功能要求：

- 支持从任务、证据、异常、依赖和验收失败中生成 review 记录。
- 系统自动识别受影响的上游目标、小目标、假设、约束和行动计划。
- 支持发起重新讨论、重新确认目标、调整资源、重排任务或重新拆解。
- Review 过程必须记录参与的人和 Agent、输入信息、推理过程、结论和变更范围。
- Review 结论必须回写工作图谱，形成新的版本关系和反馈回流关系。

验收标准：

- 任意 blocked 或 rejected 任务都必须能说明是否影响上游目标。
- Review 必须关联触发它的下游节点和受影响的上游节点。
- Review 后如果调整计划，系统必须保留调整前后的链路差异。
- 没有完成 review 的重大阻塞不能被简单关闭。

## 9. 权限与治理

### 9.1 权限模型

- 目标创建权限。
- 目标确认权限。
- 小目标确认权限。
- 图谱节点查看权限。
- 上游目标追溯权限。
- 任务编辑权限。
- Agent 调度权限。
- 证据查看权限。
- 知识条目查看权限。
- 关注列表查看权限。
- 决策待办处理权限。
- 企业系统工具调用权限。

### 9.2 权限展示规则

- 工作图谱默认只返回用户可见节点和关系。
- 用户向上追溯遇到无权限节点时，系统只提示“上游存在权限边界”，不展示具体标题、内容、责任人或证据。
- 关注的工作、决策待办和知识库必须继承图谱节点与证据来源权限。
- Agent 推荐、总结、执行和问答必须遵守当前用户、任务和工具授权范围。
- 权限变化后，相关缓存、关注列表、搜索结果和 Agent 上下文必须同步刷新。

### 9.3 审计要求

必须审计：

- 目标版本变更。
- 确认门操作。
- Agent 工具调用。
- 任务状态变更。
- 证据上传、删除、修改。
- 重大偏差处理。
- 图谱节点查看和展开行为。
- 无权限上游追溯请求。
- 知识库搜索、查看和导出。
- 权限授权、撤销和继承变化。

## 11. 关键页面

### 11.1 工作图谱

展示：

- 用户权限内的整个工作图谱。
- 当前节点、可展开子节点和可见上游路径。
- 当前节点的来源链路、服务的更大目标和下游拆解结果。
- 无权限上游节点的边界提示。
- 节点详情、关系说明、最近更新和关联操作。
- 从节点进入关注、决策待办、知识库和执行动作的入口。

### 11.2 关注的工作

展示：

- 用户收藏和关注的目标、节点、任务、证据、异常和知识。
- 系统推荐的重要工作。
- 最近变化、风险等级、提醒状态和关联负责人。
- 置顶、取消关注、调整提醒和回到图谱上下文的操作。

### 11.3 决策待办

展示：

- 待用户确认、取舍、授权或处理的事项。
- 待办来源节点、影响范围、证据依据和 Agent 建议。
- 决策选项、截止时间、风险提示和操作记录。
- 已处理事项及其对图谱的后续影响。

### 11.4 知识库

展示：

- 用户权限内的材料、证据、背景、复盘和外部系统链接。
- 与图谱节点的关联关系。
- 搜索、筛选、标签、来源追溯和可信度。
- 无权限知识的边界提示。
- 从知识条目回到相关图谱节点的入口。

### 11.5 目标创建与理解页

展示：

- 自然语言输入框。
- 背景材料上传。
- 目标类型选择。
- 参与 Agent 选择。
- 目标理解稿、假设、约束、待澄清问题和确认门。

## 14. 用户体验原则

- 面向 CEO 展示结论、关键假设和确认点，不展示过多任务细节。
- 面向执行者展示任务、依赖、证据和验收标准。
- 第一个 Tab 必须是用户权限内的整个工作图谱。
- 第二个 Tab 承载收藏、关注和重要工作。
- 第三个 Tab 只承载真正需要用户决策的事项。
- 第四个 Tab 承载权限内知识库。
- 用户向上追溯但无权限时，只展示权限边界，不展示敏感内容。
- Agent 讨论要可折叠，默认展示结构化结论。
- 大段自然语言只作为输入和解释，不作为最终工作载体。
- 文档只作为图谱的阅读视图、导出快照或汇报格式，不作为产品内的一等工作对象。
- 用户的主要操作对象是目标、假设、小目标、任务、证据、验收和异常这些图谱节点。
- 关键推理必须可追溯到 `f(x)`，让用户知道当时基于什么信息、经历什么推理过程、形成什么阶段结论。
- 任何工作单元都必须展示来源、服务的上级目标和下游影响。
- 下游失败或阻塞必须能回流到上游 review，而不能只作为局部任务状态存在。
- 所有任务都要能回溯到小目标和总目标。
- 所有完成都要有证据。

## 15. 风险与应对

### 15.1 Agent 理解错误

应对：

- 设置目标确认门。
- 保留原始表达。
- 展示待澄清问题。
- 支持版本回滚和差异对比。

### 15.2 目标拆解不合理

应对：

- 设置小目标确认门。
- 展示支撑逻辑。
- 标记高风险假设。
- 支持多方案对比。

### 15.3 执行结果不可信

应对：

- 强制证据回填。
- 支持证据来源追溯。
- 对关键任务启用人工验收。

### 15.4 产品过于复杂

应对：

- 用户入口心智保持四个 Tab：工作图谱、关注的工作、决策待办、知识库。
- 底层工作模型仍保持目标决策层和任务执行层，避免暴露过多内部结构。
- 默认隐藏内部推理细节。
- 只在确认、异常和复盘时展开推理依据。

### 15.5 权限边界造成理解断裂

应对：

- 对无权限上游节点给出明确的边界提示。
- 在不泄露敏感信息的前提下说明当前图谱是“我可见的工作图谱”。
- 提供权限申请或联系拥有者的入口。
- Agent 总结必须说明其结论基于当前可见上下文。
