## 2. 产品概述

WorkGraph 是一套以目标为根的人-Agent 协作办公系统。它将管理者的自然语言战略意图转化为可确认的目标对象，再拆解为小目标、任务、执行状态、证据和验收结果，最终形成从决策到执行的闭环。产品不以文档作为核心产物，而以可推理工作图谱作为组织工作的第一性载体。

产品一级结构采用两层：

- 目标决策层：负责意图理解、目标澄清、小目标拆解和人类确认。
- 任务执行层：负责任务生成、Agent 调度、执行推进、证据回填、验收判断和异常回流。

系统内部保留三种推理状态：

- 隐性推理：承载自然语言、直觉、类比、偏好和模糊判断。
- 显性推理：沉淀目标、假设、约束、充分条件和小目标。
- 可执行推理：生成任务、依赖、状态、工具调用、证据和验收。

## 3. 产品目标

### 3.1 用户目标

- CEO 能用自然语言提出战略目标，并快速获得结构化目标理解稿。
- 管理者能确认目标和拆解路径，而不是亲自维护所有任务细节。
- Agent 能围绕同一个目标模型协作，而不是各自生成孤立建议。
- 执行团队能看到每个任务为什么存在，以及如何证明完成。
- 组织能追踪目标从提出、确认、拆解、执行到验收的全过程。

### 3.2 产品目标

- 将战略目标转化为结构化、可确认、可执行的工作图。
- 通过确认门建立人类授权边界。
- 通过证据链提高 Agent 执行可信度。
- 通过异常回流让执行结果反向修正目标和假设。

### 3.3 非目标

- v0.1 不做完整企业 IM 替代。
- v0.1 不做通用项目管理工具的全部能力。
- v0.1 不承诺自动替代高管决策。
- v0.1 不追求全行业通用模板，优先验证战略目标落地场景。
- v0.1 不把方案文档、会议纪要或复盘文档作为核心工作产物，文档仅作为图谱导出和阅读视图。

## 4. 核心概念

### 4.1 目标 Goal

由管理者提出并确认的组织意图，包含对象、指标、时间、范围、约束和成功标准。

### 4.2 小目标 Sub-goal

支撑总目标达成的阶段性目标或充分条件。小目标应能解释其如何支撑上级目标。

### 4.3 假设 Assumption

目标拆解所依赖但尚未完全验证的前提。

### 4.4 约束 Constraint

执行过程中不可违反的条件，如预算、时间、合规、品牌、安全、资源上限等。

### 4.5 任务 Task

为达成小目标而生成的可执行单元，包含责任方、依赖、状态、验收标准和证据要求。

### 4.6 证据 Evidence

证明任务完成或目标推进的材料，可以是数据、文件、系统日志、链接、用户确认、实验结果等。

### 4.7 确认门 Approval Gate

需要人类负责人明确确认的关键节点，包括目标确认、小目标确认和重大偏差确认。

### 4.8 可推理工作图谱 Reasonable Work Graph

系统中的第一性工作产物，由节点和关系构成：

- 节点：原始意图、目标、假设、约束、小目标、任务、证据、验收、异常、决策记录。
- 关系：支撑、依赖、因果、版本、确认、阻塞、证据证明、异常影响。

文档不是一等对象。方案、纪要、周报、复盘报告只能从图谱中生成，作为人类阅读视图、对外汇报快照或归档格式。

### 4.9 推理过程 Reasoning Process

推理办公中的关键推理可以表达为：

```text
f(x) -> y
```

- `x`：当时获得的信息，包括数据、证据、背景、约束、假设和上下文。
- `f(x)`：从输入信息到阶段结论的完整推理过程，包括前提选择、证据比较、约束处理、推理步骤和结论形成。
- `y`：阶段性推理产物，包括目标草案、拆解方案、任务选择、资源配置或偏差处理方案。

系统需要记录每次关键推理的 `f(x)` 过程。复盘时，组织不只判断结果好坏，还要判断输入信息是否充分、前提是否成立、推理步骤是否跳跃、结论是否被证据验证。长期积累后，企业的推理过程可以持续校准。

## 5. 用户角色

### 5.1 CEO / 最终决策者

权限：

- 创建公司级目标。
- 审阅目标理解稿。
- 确认或驳回目标版本。
- 确认关键小目标拆解。
- 处理重大偏差。

关注点：

- 系统是否正确理解战略意图。
- 拆解路径是否符合经营判断。
- 执行是否真实推进目标。

### 5.2 目标负责人

权限：

- 承接目标。
- 补充约束、资源和背景。
- 确认非公司级小目标。
- 调整任务优先级。
- 处理执行阻塞。

### 5.3 Agent 编排者

可以是系统角色或高级用户。

权限：

- 配置参与目标讨论的 Agent。
- 指定 Agent 能访问的工具和数据。
- 调整 Agent 分工和任务边界。

### 5.4 专项 Agent

职责：

- 参与目标理解和讨论。
- 提出假设、风险、拆解建议。
- 执行分配任务。
- 回填证据和异常。

### 5.5 普通协作者

职责：

- 查看与自己相关的任务。
- 提供人工输入和确认。
- 上传证据。
- 处理 Agent 无法完成的任务。

## 6. 核心流程

### 6.1 主流程

```text
CEO 输入战略意图
-> Agent 理解目标
-> 多 Agent 讨论并形成目标理解稿
-> CEO 确认目标
-> Agent 拆解小目标
-> CEO 或目标负责人确认小目标
-> Agent 生成任务图
-> Agent / 人执行任务
-> 回填证据
-> 系统验收任务与小目标
-> 异常回流或目标达成
```

### 6.2 决策层流程

```text
意图输入
-> 目标澄清
-> 假设识别
-> 约束识别
-> 成功标准定义
-> 目标理解稿生成
-> 人类确认
-> 小目标拆解
-> 小目标确认
```

### 6.3 执行层流程

```text
小目标输入
-> 任务生成
-> 依赖分析
-> 责任分配
-> 执行调度
-> 状态更新
-> 证据回填
-> 验收判断
-> 异常回流
```

## 7. 信息架构

### 7.1 顶层导航

- 战略板：展示目标决策层，包括目标、假设、约束、小目标和确认门。
- 行动板：展示任务执行层，包括任务图、状态、依赖、证据和异常。
- Agent 协作室：展示 Agent 讨论、角色分工和输出依据。
- 证据库：沉淀所有任务和目标相关证据。
- 复盘中心：基于目标、任务、证据和结果生成复盘。
- 图谱视图：展示目标、假设、小目标、任务、证据、异常和确认记录之间的可推理关系。

### 7.2 目标详情页

模块：

- 原始意图。
- 目标理解稿。
- 目标字段。
- 假设列表。
- 约束列表。
- 成功标准。
- 小目标树。
- 确认记录。
- 关联任务。
- 执行进度。
- 风险和异常。

### 7.3 任务详情页

模块：

- 任务描述。
- 支撑的小目标。
- 责任 Agent / 人。
- 依赖任务。
- 输入与输出。
- 执行状态。
- 验收标准。
- 证据列表。
- 异常记录。

## 8. 功能需求

### 8.1 目标创建

用户故事：

作为 CEO，我希望用自然语言输入战略目标，使系统自动理解并生成结构化目标草案。

功能要求：

- 支持自然语言输入目标。
- 支持上传背景材料，如市场报告、经营数据、竞品分析。
- 支持选择目标类型，如营收目标、增长目标、市场进入、产品发布、降本增效。
- 系统自动抽取关键字段：市场、指标、目标值、时间周期、业务范围、约束、负责人。
- 系统保留原始表达，不覆盖人类意图来源。

验收标准：

- 输入一句自然语言战略目标后，系统能生成结构化目标草案。
- 草案必须展示缺失字段和待澄清问题。
- 原始输入必须可追溯。

### 8.1.1 图谱化原则

功能要求：

- 所有目标创建结果必须进入可推理工作图谱。
- 系统不得只生成一份静态目标说明文档作为产物。
- 目标理解稿应表现为目标节点、假设节点、约束节点、成功标准节点和待确认问题节点。
- 所有自然语言说明必须能映射到图谱节点或作为节点备注存在。

验收标准：

- 目标创建完成后，系统必须生成图谱节点和关系。
- 用户可以从图谱生成目标说明文档，但不能只保存文档而不保存图谱。
- 图谱节点更新后，相关文档视图必须能重新生成或提示过期。

### 8.2 目标理解与澄清

用户故事：

作为 CEO，我希望 Agent 在提交目标前先充分讨论并列出问题，使我确认的是清晰目标而不是模糊口号。

功能要求：

- 多 Agent 根据各自角色对目标进行分析。
- 系统生成澄清问题，如时间范围、口径、资源边界、风险偏好。
- 系统识别隐含假设。
- 系统识别可能冲突的约束。
- 系统输出目标理解稿。

目标理解稿字段：

```yaml
goal:
  raw_expression: ""
  normalized_description: ""
  metric: ""
  target_value: ""
  market_scope: ""
  time_range: ""
  owner: ""
  assumptions: []
  constraints: []
  success_criteria: []
  open_questions: []
```

验收标准：

- 目标理解稿必须包含目标字段、假设、约束、成功标准和待确认问题。
- Agent 讨论结论必须能追溯到参与 Agent 的观点。
- CEO 可以对字段逐项修改或确认。

### 8.3 目标确认门

用户故事：

作为 CEO，我希望在系统推进执行前确认目标版本，避免 Agent 误解战略意图。

功能要求：

- 支持确认、驳回、要求补充澄清。
- 目标确认后生成不可变版本号。
- 所有后续小目标和任务必须关联目标版本。
- 目标变更时系统提示影响范围。

验收标准：

- 未确认目标不能进入任务执行层。
- 目标确认记录包含确认人、时间、版本和备注。
- 目标变更后必须生成新版本。

### 8.4 小目标拆解

用户故事：

作为目标负责人，我希望系统把总目标拆解为若干小目标，并说明每个小目标如何支撑总目标。

功能要求：

- Agent 基于目标、假设和约束生成小目标树。
- 每个小目标必须包含支撑逻辑。
- 支持按时间、业务模块、漏斗环节、组织职能、充分条件等方式拆解。
- 支持展示小目标之间的依赖关系。
- 支持标记关键小目标。

小目标字段：

```yaml
sub_goal:
  id: ""
  description: ""
  parent_goal_id: ""
  support_logic: ""
  metric: ""
  target_value: ""
  owner: ""
  dependencies: []
  risks: []
  acceptance_criteria: []
```

验收标准：

- 每个小目标必须关联上级目标。
- 每个小目标必须有支撑逻辑。
- 系统必须标出拆解中的高风险假设。

### 8.5 小目标确认门

用户故事：

作为 CEO 或目标负责人，我希望确认关键小目标，确保执行路径没有偏离战略。

功能要求：

- 支持逐项确认小目标。
- 支持批量确认低风险小目标。
- 支持要求 Agent 重新拆解。
- 支持对小目标设置确认责任人。

验收标准：

- 未确认的关键小目标不能生成正式任务。
- 确认记录可追溯。
- 重新拆解必须保留历史版本。

### 8.6 任务生成

用户故事：

作为执行负责人，我希望系统基于确认后的小目标生成任务图，减少人工拆解成本。

功能要求：

- 基于小目标生成任务列表。
- 自动识别任务依赖。
- 自动建议责任 Agent 或人类角色。
- 自动生成验收标准和证据要求。
- 支持人工编辑任务。

任务字段：

```yaml
task:
  id: ""
  description: ""
  supports_sub_goal_id: ""
  owner_type: "agent | human | team"
  owner_id: ""
  dependencies: []
  status: "pending | running | blocked | done | rejected"
  priority: "low | medium | high | critical"
  input_requirements: []
  expected_output: ""
  acceptance_criteria: []
  required_evidence: []
```

验收标准：

- 每个任务必须关联至少一个小目标。
- 每个任务必须有验收标准。
- 每个任务必须有责任方。

### 8.7 Agent 调度与执行

用户故事：

作为目标负责人，我希望 Agent 能根据任务图自动执行任务，并在需要人工判断时发起请求。

功能要求：

- 支持将任务分配给不同 Agent。
- 支持 Agent 调用外部工具和企业系统。
- 支持任务状态自动更新。
- 支持 Agent 在遇到权限、数据缺失、判断不确定时请求人工介入。
- 支持任务执行日志。

验收标准：

- Agent 执行必须记录输入、操作、输出和时间。
- Agent 无法继续执行时必须进入 blocked 状态并说明原因。
- 关键工具调用必须可审计。

### 8.8 证据回填

用户故事：

作为管理者，我希望看到任务完成的证据，而不是只看到状态变成完成。

功能要求：

- 支持 Agent 自动上传证据。
- 支持人工上传证据。
- 支持证据类型标记：数据、文档、链接、截图、日志、审批、实验结果。
- 支持证据与任务、小目标和目标关联。
- 支持证据可信度标记。

验收标准：

- 没有满足证据要求的任务不能自动验收通过。
- 证据必须可追溯到来源。
- 证据删除或变更必须记录。

### 8.9 验收判断

用户故事：

作为目标负责人，我希望系统判断任务是否真的支撑小目标，而不是只完成了表面动作。

功能要求：

- 系统根据验收标准和证据判断任务完成情况。
- 支持自动验收、人工验收和混合验收。
- 支持验收不通过并要求返工。
- 支持小目标级验收。
- 支持总目标进度判断。

验收标准：

- 验收结果必须包含依据。
- 验收不通过必须说明缺口。
- 小目标验收必须基于关联任务和指标。

### 8.10 异常回流

用户故事：

作为 CEO，我希望执行中发现重大偏差时，系统能自动回到决策层，而不是继续错误执行。

功能要求：

- 系统识别重大偏差：目标不可达、资源不足、假设被推翻、合规风险、关键任务阻塞。
- 自动生成偏差报告。
- 发起重大偏差确认门。
- 支持调整目标、小目标、任务、资源或约束。
- 记录决策变更原因。

验收标准：

- 重大偏差不能只停留在任务层，必须回流目标决策层。
- 偏差报告必须包含影响范围和建议动作。
- 决策变更后必须更新关联任务。

### 8.10.1 推理记录与推理过程校准

用户故事：

作为 CEO 或目标负责人，我希望系统记录关键推理当时基于什么信息、经历了什么推理过程、形成了什么阶段结论，并在复盘时指出哪些推理步骤有效、哪些前提失真、哪些结论跳跃，从而持续提升组织推理质量。

功能要求：

- 关键推理必须生成推理记录节点。
- 推理记录必须包含输入信息 `x`、推理过程 `f(x)`、阶段结论 `y`。
- 输入信息应关联证据、数据、背景材料、假设、约束和上下文。
- 推理过程应记录前提选择、证据比较、约束处理、关键步骤、模型判断和人工偏好。
- 输出结果应关联目标、小目标、任务、资源配置或偏差处理方案。
- 复盘时系统应比较推理过程、阶段结论和实际结果，标记信息缺失、前提失真、推理跳跃和过程校准建议。

验收标准：

- 目标确认、小目标确认、重大偏差处理必须生成推理记录。
- 每条推理记录必须能追溯输入信息和输出结果。
- 复盘报告必须能按推理链路展示哪些 `f(x)` 过程被验证，哪些需要校准。

### 8.11 Agent 协作室

用户故事：

作为管理者，我希望看到 Agent 之间如何讨论目标和拆解方案，以便判断输出是否可信。

功能要求：

- 展示参与 Agent 列表和角色。
- 展示关键观点、分歧和结论。
- 支持将讨论内容折叠为决策摘要。
- 支持查看 Agent 输出依据。
- 支持人工追问。

验收标准：

- Agent 讨论不能只保存聊天流水，必须生成结构化摘要。
- 分歧必须被标记。
- 进入目标理解稿的结论必须可追溯。

### 8.12 复盘中心

用户故事：

作为管理者，我希望系统基于目标和执行证据自动生成复盘，帮助组织学习。

功能要求：

- 自动生成目标复盘报告。
- 对比目标值、阶段值和实际结果。
- 总结关键假设是否成立。
- 总结任务完成情况和证据。
- 总结重大偏差和决策调整。
- 输出下一轮建议。

验收标准：

- 复盘必须引用目标、任务、证据和决策记录。
- 复盘结论必须区分事实、判断和建议。

## 14. 用户体验原则

- 面向 CEO 展示结论、关键假设和确认点，不展示过多任务细节。
- 面向执行者展示任务、依赖、证据和验收标准。
- Agent 讨论要可折叠，默认展示结构化结论。
- 大段自然语言只作为输入和解释，不作为最终工作载体。
- 文档只作为图谱的阅读视图、导出快照或汇报格式，不作为产品内的一等工作对象。
- 用户的主要操作对象是目标、假设、小目标、任务、证据、验收和异常这些图谱节点。
- 关键推理必须可追溯到 `f(x)`，让用户知道当时基于什么信息、经历什么推理过程、形成什么阶段结论。
- 所有任务都要能回溯到小目标和总目标。
- 所有完成都要有证据。

## 15. 风险与应对

### 15.1 Agent 理解错误

应对：

- 设置目标确认门。
- 保留原始表达。
- 展示待澄清问题。
- 支持版本回滚和差异对比。

### 15.2 目标拆解不合理

应对：

- 设置小目标确认门。
- 展示支撑逻辑。
- 标记高风险假设。
- 支持多方案对比。

### 15.3 执行结果不可信

应对：

- 强制证据回填。
- 支持证据来源追溯。
- 对关键任务启用人工验收。

### 15.4 产品过于复杂

应对：

- 用户心智保持两层：战略板和行动板。
- 默认隐藏内部推理细节。
- 只在确认、异常和复盘时展开推理依据。
